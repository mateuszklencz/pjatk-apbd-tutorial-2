# pjatk-apbd-tutorial-2
Repo for task 2 from Computer Science major in Polish-Japanese Academy of Information Technology (PJATK) for course Database Applications . 

# Task & Requirements
## Task description
Prepare a console application in C# that supports a university equipment rental service. The system 
should allow equipment registration, user rentals, returns, availability control, and the generation of 
basic reports 

The system should include different types of equipment, with some shared properties and some 
specific ones. This is a good opportunity to think carefully about what should be common, what 
should be specialized, and where inheritance makes sense as opposed to composition or a dedicated 
service class

## Domain model - minimum scope
Your solution should include at least the following domain elements: 
- Equipment as the common concept for all device types. 
- At least three equipment types, for example `Laptop`, `Projector`, and `Camera`. 
- A system user with at least two user types, for example `Student` and `Employee`. 
- A rental object describing the rental date, due date, actual return, and resulting fee or penalty. 
- One service or a set of services responsible for business logic. 

## Required data
- Each equipment item should have a unique identifier generated by the system. 
- Each equipment item should store a name, availability status, and basic shared data. 
- Each equipment type should contain at least 2 type-specific fields. 
- Each user should have an identifier, first name, last name, and user type. 
- Each rental should store who rented what, when it was rented, for how long, and whether the 
return was on time

## Functional requirements 
The application should support at least the following operations: 
1. Add a new user to the system. 
2. Add a new equipment item of a selected type. 
3. Display the full list of equipment together with current status. 
4. Display only equipment currently available for rental. 
5. Rent equipment to a user. 
6. Return equipment and calculate a possible late penalty. 
7. Mark equipment as unavailable, for example because of damage or maintenance. 
8. Display active rentals for a selected user. 
9. Display the list of overdue rentals. 
10. Generate a short summary report of the rental service state. 

## Business rules 
- A student may have at most 2 active rentals at the same time. 
- An employee may have at most 5 active rentals at the same time. 
- If an equipment item is marked as unavailable, it cannot be rented. 
- The system should block a rental if the user has exceeded their limit. 
- A late return should result in a penalty according to a rule defined by you. 
- Penalty rules and user limits should be written in a way that is readable and easy to modify.

## Design requirements
This is the main part of the exercise. The code should not only work, but also be reasonably 
organized.  
- Do not place the whole business logic in `Program.cs` or in one large class such as `App`. 
- Separate the domain model from the execution logic and from the console interface. 
- Make sure that each class has one clear responsibility. 
- Avoid tight coupling between classes where it can be reasonably avoided. 
- If you use inheritance, it should follow from the domain model rather than from a desire to make  the solution look more object-oriented. 
- Where it makes sense, use an interface or an abstract class. 
- If an operation may fail, handle that explicitly through an exception, a result object, or another  clear mechanism. 

"Good practice: If a business rule may change, do not hard-code it in five different places. It is 
better to have one class or one clearly defined area responsible for a particular decision than to 
spread the logic across the whole project" 

## GitHub Repo Requirements
- Place the solution in your own GitHub repository. 
- The repository should contain a readable work history with at least 8 meaningful commits. 
- Use at least 2 working branches and merge them into `main` after the work is complete. 
- The repository should contain a `README.md` file with a project description and a short justification of design decisions. 
- In the `README`, indicate where the project shows an attempt to address cohesion, coupling, and  class responsibilities. 
- In the `README`, briefly justify why you chose the particular organization of classes, files, and  possible project layers.

4. An attempted invalid operation, for example renting unavailable equipment or exceeding a user 
limit. 
5. A return completed on time. 
6. A delayed return that leads to a penalty. 
7. Displaying a final report of the system stat
